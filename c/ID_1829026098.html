here i introduce to you the idea of type classes, which is a type interface system that is very common in well designed programming languages and is very popular

type classes were introduced by the haskell programming language and haskell always remained to be the testbed for new research in this topic lately popular industrial languages also started to adopt it : rust's "traits" and c++'s "concept"s are type classes under different names

from the name "type classes" : do not try to extract accurate meaning, it is just one name, that happened to be the name but is not distinctive

type classes is an alternative of the object oriented type interface system, which is defined around the core idea that all the methods in type interfaces should be late-bound, that is their dispatch should happen dynamically, that is : in run-time the fundamental difference between type classes and oo interfaces is that the methods in type classes are dispatched statically, that is : in compile time this makes a huge difference and allows type classes to be much more expressive and natural

some advantages are the following :

faster program execution

conditional implementation of interfaces becomes easily possible for example the type 'Maybe t' implements interface 'Equatable' if and only if its element type 't' implements interface 'Equatable' this capability is very important because working with container types [like Maybe, Result, List, newtype wrappers] is ubiquitous

we can have methods without input this capability is important because many interfaces have such methods, for example the important interfaces 'Default' and 'Monoid'

methods do not need to have a special input, hence cases when they conceptually do not have special input are expressed more naturally for example in the interfaces Equatable and Comparable the compare methods have 2 inputs that to be compared the roles of these inputs are the same but their expression in java for example contains a big difference in the 2 inputs, one of the inputs is implicitely, the other is explicitely written this suggests to the reader of the code that the 2 inputs have different roles, which is false such mismatch between the code and the conceptual reality makes the code unnecessarily difficult to read and programming in the language unnecessarily confusing to learn

static dispatch does not prevent the ability to dynamic dispatch in the rare cases when that is needed

type classes are very popular and it looks that in the future they will be part of most well designed, practical programming languages

paradigm, but a paradigm should be generally applyable oo interfacing is not generally applyable it is too ugly and too weak as you saw from the examples : not only it can not express such basic type interfaces as Equatable and Comparable in a natural way, but even if we accept its ugliness : so soon as wrapping our [now equatable or comparable] type by a Maybe or other container : the oo type interfacing can not follow us and for interfaces with input-less methods, like the very important Monoid interface : it can not even start

one of the roots of the failure of oo is one of its foundational principle that method dispatch should be dynamic in the world of type interfacing : this is corrected by type classes
