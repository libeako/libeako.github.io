in software coding

computer programs

imperative

an imperative program can be thought of as a list of instructions

for ergonomical, practical reasons we often like to treat a sequence of related instructions as a single, "composed" instruction, give name to it we make our decisions about these namings based on our intuition, understanding the domain we are working in in other words : we want the composition to be driven by the busieness domain and be free of technical constraints

because imperative programs are lists of instructions : imperative program composition is list concatenation

the associative nature of list concatenation is exactly the reason why we may "compose" instructions into subprograms

functional

similarly to the imperative case the parts of a functional program are functions hence the composition of the parts is function composition, which is associative

without composition we would have to work like in an assembly programming language

thus composability, that is associativity is fundamental for software coding

a lot of abstraction in software coding is composable people even consciously try to design their abstractions to be composable
