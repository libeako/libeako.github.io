type classes are very popular in haskell and lately popular industrial languages also started to adopt it : rust's "traits" and c++'s "concept"s are type classes under different names

it is a "class" of types you can think of it as its definition contains a set of requirements about the types that belong to it the coder may make a type to implement a type class by declaring the fact of implementation and the way how each requirement in the type-class is met by the type

from the name "type classes" : do not try to extract accurate meaning, it is just one name, that happened to be the name but is not distinctive

type class in idris is named "type interface" i may also use "interface" instead of "class" sometimes

type classes do not increase the expressive power of the language, do not enable additional abstractions they only make the coding work more convenient, less verbose

a type class can be seen as all of these

interface for types, similarly to as in java, c#

partial function from the set of types

as special case : mapping types to their default values

via this the language can provide the ability of implicit arguments to functions

a principled method overloading technique

definition of a type interface

a simplest example for start

interface Equatable t equal : t -> t -> bool

"interface" is a keyword here

haskell uses the "class" keyword

idris uses the "interface" keyword

"Equatable" is the name of the type interface

the rest of the lines list the (name : type) pairs that to each the interface-implementation has to provide a value of

"t" is the name of the unknown type to implement the interface

this is a a significant difference compared to java and c# giving a name to the implementing type makes us able to refer to it, everywhere, conveniently

advantages

the Equatable example in c#

interface IEquatable { bool Equals(??? other); }

what should go in place of "???" ?

in java and c# this problem is solved with "F-bounded polymophism", which i consider a trick

interface IEquatable<T> { bool Equals(T other); }

type polymorphism is used to smuggle the name of the type that implements the interface into the interface definition

the interface may contain 0-ary functions

a useful simple example

interface Default t def : t

nullary functions are impossible to be in java and c# interfaces, not even with the f-bounded polymorphism trick but nullary functions are parts of some very important practical interfaces for example : Monoid, Applicative, Monad

making the example more elegant by using infix function symbol

interface Equatable t (==) : t -> t -> bool

the interface definition may contain multiple requirements

interface Equatable t (==) : t -> t -> bool (/=) : t -> t -> bool

it is possible to specify default implementations for the requirements

interface Equatable t (==) : t -> t -> bool (/=) : t -> t -> bool x == y = not (x /= y) x /= y = not (x == y)

even mutually recursive ones

implementation of a type interface

a simple example

implements Equatable Int (==) = Integer.equality

"implements" is a keyword here

haskell uses keyword "instance"

(==) of Equatable is defined by this implementation as the eqution test provided by module 'Integer'

(/=) [not equal] does not have to be implemented, because interface Equatable contains a suitable default implementation of it

(type T implements interface I) as a piece of code is not fixed to the definition of type T it can be also in the module of the interface

this makes us possible to implement new interfaces for types without modifying the definition of the type this is in sharp contrast with the java and c#, where the type definition must declare the list of implemented interfaces

the implementation can be conditioned

a big problem exists for defining interface implementations for collection types [tuples, Array, List, Maybe, Either, wrapper types, ...] very often they can implement an interface sensibly iff the element type implements that interface to solve this problem the programming language must provide the ability to condition the implementation

example

implements (Equatable a, Equatable b) => Equatable (a * b) (x1, y1) == (x2, y2) = x1 == x2 && y1 == y2

at the source side of the fat arrow is the condition

implicit arguments

functions may take type interface implementations as implicit input for example :

minimum : Ordered e => List e -> Maybe e

function 'minimum' takes an implicit parameter specifying how type 'e' implements interface 'Ordered'

comparison with object oriented type interfacing

the main difference between the object oriented type interfaces and the haskell-style type classes is in how the calls for the methods of the interfaces are dispatched in OO : dynamically, in type classes : statically in OO an instance is needed, which contains a pointer to its type - that is the type is discovered in run-time with type classes the type is known at compile-time

the static solution is much better, as i hope i will convince you

dynamic method dispatch is needed very rarely

this may surprise you if you are used to badly disigned object-oriented language like java and c#, which force the code to use their type interfacing solution with dynamic dispatch machinery, even when simpler but not provided tools, like sum types or higher order functions would suffice once someone gets to use a well-designed language like haskell : it quickly realizes that dynamic method dispatch is needed very rarely

and when needed it is not difficult to get it for example in Haskell with just a little more verbosity than in java

with the static solution it is easy to provide the ability to implement an interface conditionally for example : that type 'Maybe t' implements the equatable interface if and only if its content type t implements the equatable interface

this is very important because such type modifiers, as Maybe, List, newtype wrappers, ... are ubiquitous a type interface system that breaks down already at meeting these can not be considered good enough servants of the coder hence in languages like java and c# that do not provide conditional interface implementation their type interfacing solution [also known as "object orientation"] can not be considered viable software coding paradigms

with the dynamic solution it is not possible to have nullary methods [methods without any input] but some very important type interfaces, like default and monoid have such methods

the static solution does not require a special implicit this input hence methods that do not have such input feel much more natural with it for example the methods of the equatable and comparable type interfaces - these compare 2 values of the same type that have the very same role - this role sameness is shown by the static solution, why the dynamic solution shows the false suggestion that one of the inputs is special - such mismatch between the programming language and the conceptual reality make the learning process of new programmers significantly more difficult

this is the foundation of that the haskell type interface mechanism is much faster and much more expressive and usable and more natural than java's and c#'s object-oriented "interface"s

laws

laws are intended mathematical properties [usually equalities] of interface implementations that though can not be stated and checked by static typing cheaply enough, hence they are written only in comment or documentation example : the laws of interface Equatable should be the usual mathematical properties of equality: equivalence, substitutivity it is a very important convention that implementations of an interface must comply with the laws of that interface not all interfaces have to have laws examples for interfaces that do not have laws : Default, Foldable [you may know this as "Enumerable"]
