most functional programming languages adopt the simplicity of math : a function has exactly 1 input and exactly one output this is a very elegant and practical simplicity

how are then functions with multiple inputs represented ?

the intuitive solution is to pack the multiple inputs into one with product type construction

x1 * x2 -> y

"currying" is an other solution

x1 -> (x2 -> y)

in natural language : a 2 parameter function is a function from the first parameter to a function that takes the second parameter

the "->" type operator is right associative so we may drop the brackets for elegance

x1 -> x2 -> y

sometimes this is named "the curried form" of (x1 * x2 -> y)

converting between these 2 usual representations of input functions is a common task

common names for the converting functions

curry : (a * b -> c) -> (a -> b -> c) uncurry : (a -> b -> c) -> (a * b -> c)

implementing these and prooving that they are inverse of each other should be an easy excercise

the name "currying" comes from the fact that Haskell Curry was one of the first who used the curried form though he stated that he learned it from Schönfinkel

if the order of the inputs of a function is not what we want at a certain application site then one can use the common input-order-flipping function

flip : (a->b->c) -> (b->a->c)
