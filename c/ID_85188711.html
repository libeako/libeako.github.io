type Either L R = | left L | right R

as you can see : the 'Either' type is mathematically equivalent to the 'Result' type

'Either' vs 'Result'

one if not the most common usage of such a sum type is to represent result of computations that can fail 'Result's naming is designed specifically for this this spcecificity makes it ergonomic for this usage, but is too awkward in every other case

the naming of 'Either' is general, it does not suggest any particular usage scenario it is less readable as a result of computation, but it can be used for anything without feeling awkward

which one of them should we use ? :

using 'Result' is too awkward for non-result cases

using 'Either' is less readable for result cases, which is the a very important use-case

using them both would require conversions between them

the communities of the Haskell and similar languages chose to use 'Either' always [there is not a separate 'Result'] they are satisfied with this choice after tens of years of practice

the key in this decision is that the 'Either' type is more suitable to represent computation results than one may feel at first sight the cause of this is that the choice between the sides in the 'Either' type is much less ambiguous than one usually feels at first the "right" side in 'Either' should be used for the successful case the reasons of this are :

luck from language English :

the word "right" in means both a side and a positive judgment, which fits well for the success case of a result : the "right" result is the good one, that is the successful one

by the order of parameters :

this is out of the scope of this topic, hence you do not need to understand it this is here only for the sake of completeness of the argument

semantically :

a Result type, when the failure type parameter is fixed is an instance of the very important 'Monad' type class with the [failure, success] parameter order [and only with this] the fixing of the failure type parameter is simply a partial [type-]function application : 'Result failure'

syntactically :

the usual writing order is left -> right the parameterization order "Either l r" fits to it

these 2 considerations imply that left = failure, right = success
