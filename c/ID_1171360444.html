is an application of a function to less actual inputs than the number of its formal inputs

then the result of the function application expression is a function that expects actual inputs for the rest of the formal inputs

formally :

f : a -> b -> c

=>

f x : b -> c

function application is left associative

examples

f x y = (f x) y f x y z = ((f x) y) z

this is in nice consistency with partial function application, because by this (f x) is a sub-expression of (f x y) by this : partial function application is not a special language construct, but a natural capability, which is awesome

example

plus : Number -> Number -> Number plus x y = x + y increment = plus 1

increment could also have been written as follows

increment x = plus 1 x

but dropping input x from both sides of the defining "=" leads to a more short, readable, elegant code, at least in this simple case

order of the inputs

to enable partial function application in greatest scope : function inputs are often ordered backwards to what imperative programmers are used to

examples

map : (a -> b) -> List a -> List b filter : (a -> Bool) -> List a -> List a

in the functions 'map' and 'filter' the order of input parameters is opposite of what an imperative programmer may find intuitive the advantage of this "reverse" order is hopefully demonstrated by the following example

filter_then_map

filter_then_map : (a -> bool) -> (a -> b) -> List a -> List b

for the definition of 'filter_then_map' the following are equivalent to each other

filter_then_map p f l = map f (filter p l) filter_then_map p f l = ((filter p) >>> (map f)) l filter_then_map p f = filter p >>> map f
